How a Production AI System Enforces Quality

Live examples drawn from the RARIX codebase (rarix repo)

🧪
Hybrid Test Runners
Vitest 4.1.6 + node:test — 621 test files

Vitest for ESM/async modules; node:test for pure functions. Two runners coexist for a reason: each is optimised for a different part of the module graph. Neither is a compromise.

🛡️
TDD Enforcer Hook
PreToolUse hook

Blocks Write/Edit without a matching test file. 5-minute override window for refactors. Test-first is enforced, not optional. Agents cannot write implementation before writing tests.

🔬
Mutation Testing
Stryker 9.6.1

Thresholds: 80 high / 60 low / 50 break. The CI gate is mutation score, not line coverage. A test suite that hits every line but catches no mutations is treated as untested.

🗄️
Anti-Pattern Catalog
SQLite-backed

Each PR fix captures pattern + ripgrep detector regex — pre-commit blocks it forever. Learning loop: bugs caught in review become automated gates for every future PR.

👁️
7-Dimension Code Review
Weighted dimensions

Correctness 25% · Security 20% · Performance 10% · Readability 15% · Maintainability 10% · Test Coverage 10% · Architecture 10%. Each dimension scored independently, evidence-quoted, and aggregated.

🎭
Playwright MCP
@playwright/mcp@latest

Registered as high-priority MCP server. Accessibility tree (not screenshots). 4x more token-efficient than vision models. Works across Chromium, Firefox, and WebKit.

📊
Protocol Compliance Ledger
SQLite event log

Every skill execution scored. Results queryable. Compliance is a metric, not a feeling. Agents cannot self-report compliance — it is recorded externally and auditable.

🔒
PreToolUse Gates
43 hook files

Blocks: dangerous commands, SQL DROP, secrets exposure, cost budget overrun, branch prefix violations, constitution checks. Hooks run before the tool executes — damage prevention, not damage repair.

📈
TDD Compliance Tracker
Per-commit metric

Rule: test-lines ≥ 1/3 src-lines. Tracked per agent per commit. JSON output for dashboards. Non-compliance is visible, trending, and used in agent performance evaluation.

🚦
CI Pipeline
5 parallel jobs

Lint · Tests (1,252) · Sub-packages (543 hook tests, REQUIRED gate) · Architecture · Semgrep static analysis. All five jobs must pass. Sub-package failures block merge even if top-level tests pass.

3e — Property-Based Testing

fast-check

QuickCheck for JavaScript/TypeScript. Works with Vitest and Jest. Finds shrinkable counterexamples — when a failure is found, it automatically minimises the failing input to the simplest possible case. Source: fast-check.dev

Hypothesis (Python)

Gold standard for property-based testing. 2025 NeurIPS study: Claude autonomously discovered real bugs in numpy and pandas by writing Hypothesis tests derived from type annotations and docstrings alone.

Key concept: Describe invariants ("total after discount ≥ 0"), let the framework generate thousands of inputs automatically. You define what must always be true; the framework tries to prove you wrong.

— Property-Based Testing core principle

3f — Mutation Testing

Stryker (JS/TS): 5,000 mutants/hour. Thresholds: 80% high / 60% low / 50% break (CI fail). HTML reports with per-mutant drill-down. Integrates with Vitest and Jest.

Key Insight
"100% coverage + 4% mutation score = 96% of bugs undetected.
Coverage tells you which lines ran. Mutation testing tells you if your tests would actually catch a bug."

AI + Stryker (2025): Outsight AI case study — feeding surviving mutants back to Cursor improved mutation score from 70% to 78%. Meta presented LLM + mutation testing work at FSE 2025, showing LLMs can generate targeted tests for surviving mutants more effectively than random test augmentation.

When AI Tests Lie: 4 Failure Modes

Understanding how LLM-generated tests fail — and what to do about each pattern

Existence-Only Assertions

Danger

The test checks that a field exists but never validates its value. Passes even if the value is wrong, zero, null, or completely different from the expected result. The assertion gives false confidence.

it('returns product', () => {
  const p = getProduct(1);
  expect(p).toBeDefined();        // ← only checks not undefined
  expect(p.price).toBeDefined(); // ← never checks p.price > 0
});
Fix

Assert specific values: expect(p.price).toBe(29.99) or at minimum expect(p.price).toBeGreaterThan(0). If you don't know the exact expected value, that's a design signal — define it first.

Over-Mocking

Danger

AI aggressively mocks dependencies and then asserts the mock's own return values. The test verifies that the mock works as configured — not that the code under test produces the right result. Real behaviour is never exercised.

it('calculates total', () => {
  const mockCart = {
    items: [{ price: 10 }],
    total: vi.fn().mockReturnValue(10)
  };
  // ← testing the mock, not the code
  expect(mockCart.total()).toBe(10);
});
Fix

Test real implementations. Only mock at true system boundaries: network calls, database I/O, external APIs, and time. If your unit has no I/O, it should have no mocks.

Coverage Without Quality

Warning

AI optimises for hitting every line and branch. Achieves 85%+ line coverage. Mutation score: 20%. 80% of potential bugs remain undetected by a test suite that looks comprehensive on paper.

"LLM-generated tests achieve only 20.32% mutation scores on complex real-world functions, even when line coverage exceeds 80%." — Outsight AI, 2025

// Coverage tool reports: 92% ✓
// Stryker reports: 18% mutation score ✗
// 82% of bugs would go undetected
it('runs the branch', () => {
  applyDiscount(100, 0.1);  // ← no assertion
});
Fix

Always run mutation testing on AI-generated test suites. Target 70%+ mutation score for critical paths. Coverage metrics are necessary but not sufficient — treat them as a floor, not a ceiling.

Oracle Failures

Warning

AI validates "what your code does" not "what it should do." When the implementation contains a bug, AI observes the buggy output, treats it as correct, and writes a test that documents the bug as expected behaviour.

"Prompting strategy matters more than model choice for oracle quality. Models shown working code before writing tests document bugs as features." — ASE 2025

// Implementation has off-by-one error
// Correct result should be 90.00
it('applies 10% discount', () => {
  // AI observed buggy output, locked it in
  expect(applyDiscount(100, 10)).toBe(91);
});
Fix

Write the test before the implementation (TDD). When AI writes tests post-implementation, always manually verify expected values against the specification, not against the code output. Specification must be the oracle, not the implementation.
